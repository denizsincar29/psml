# PSML — Prompt String Markup Language

Псевдо-HTML для описания промпта шелла. Пишешь разметку — конвертер
превращает её в готовую escape-строку (или функцию/замыкание — зависит от
шелла) для конкретного шелла.

> **Про rust-порт**: этот файл — описание самого языка PSML, взятое из
> оригинального python-проекта (`psml.py`), синтаксис не менялся. Rust-порт
> в этом репозитории умеет больше шеллов, чем python-версия (bash, zsh,
> fish, PowerShell, cmd.exe, Nushell — полная таблица и оговорки по
> каждому в [`README.md`](./README.md#поддерживаемые-шеллы)). Там же, где
> ниже упоминается `python3 psml.py`, для rust-сборки читай
> `./target/release/psml`.

```bash
python3 psml.py example.psml --shell bash
# PS1='...'
```

## Структура документа

```html
<!doctype psml>                 <!-- необязательно, просто игнорируется -->
<psml shell="bash">             <!-- shell необязателен, по умолчанию bash -->
  <head>
    <title>Заголовок окна терминала</title>
  </head>
  <body>
    ... сама разметка промпта ...
  </body>
</psml>
```

`shell` можно не указывать в файле и передать флагом `--shell` — флаг
командной строки главнее атрибута в файле. В python-версии значение
ограничено `bash`/`zsh`; в rust-порте — любой ключ из таблицы шеллов
(`psml --list-shells`).

`<title>` поддерживает только текст — без вложенных тегов (в rust-порте это
явно проверяется на этапе разбора; смысла вкладывать туда стиль/цвет нет —
заголовок окна терминала их не отображает).

## Теги

### Переменные (самозакрывающиеся)

| Тег | Значение |
|---|---|
| `<user/>` | имя пользователя |
| `<host/>` | короткое имя хоста |
| `<hostfull/>` | полное имя хоста (FQDN) |
| `<cwd/>` | текущая директория (`$HOME` → `~`) |
| `<cwdbase/>` | только последний компонент пути |
| `<symbol/>` | `$` для пользователя / `#` для root |
| `<jobs/>` | число фоновых job'ов |
| `<exitcode/>` | код возврата последней команды |
| `<time mode="24\|12\|ampm\|24short"/>` | текущее время |
| `<date fmt="%Y-%m-%d"/>` | дата (`fmt` необязателен, формат strftime) |
| `<git prefix=".." suffix=".."/>` | имя текущей git-ветки |
| `<cmd run="команда"/>` | вывод **любой** shell-команды (см. ниже) |
| `<reset/>` | сброс всех цветов/стилей |
| `<br>` / `<br/>` | перевод строки |

### Стилевые (контейнерные, можно вкладывать друг в друга)

| Тег | Значение |
|---|---|
| `<bold>...</bold>` (= `<b>`) | жирный |
| `<underline>...</underline>` (= `<u>`) | подчёркнутый |
| `<italic>...</italic>` (= `<i>`) | курсив (не все терминалы умеют) |
| `<color fg=".." bg="..">...</color>` | цвет текста / фона |

Цвет — это имя (`red`, `brightblue`, `gray`...), число `0-255` (256-цветный
режим) или `#rrggbb` (24-бит truecolor).

Пример вложенности:

```html
<color fg="green"><bold><user/>@<host/></bold></color>
```

## Как добавить вывод произвольной команды (например, git status)

В языке есть готовый шорткат `<git/>` (показывает имя ветки), но если нужно
что-то ещё — статус, venv, kubectl-контекст, батарейка, — используй
`<cmd run="...">`. Атрибут `run` — это shell-команда, её вывод подставится
прямо в промпт:

```html
<color fg="red"><cmd run="git status --porcelain 2&gt;/dev/null | grep -q . &amp;&amp; printf '*'"/></color>
```

Это покажет красную `*`, если в репозитории есть незакоммиченные изменения,
и ничего — если всё чисто. Так можно прицепить что угодно, например:

```html
<!-- активный python venv -->
<cmd run="[ -n &quot;$VIRTUAL_ENV&quot; ] &amp;&amp; printf '(%s) ' &quot;$(basename $VIRTUAL_ENV)&quot;"/>

<!-- цвет exit-кода: красный если команда упала -->
<cmd run="[ $? -ne 0 ] &amp;&amp; printf '\033[31m'"/>
```

**Важный момент про `&lt;`, `&gt;`, `&amp;`, `&quot;`:** атрибут `run` живёт
внутри HTML-тега, поэтому символы `<`, `&` и кавычку, которой обёрнут сам
атрибут, нужно экранировать HTML-сущностями (`&lt;` `&gt;` `&amp;` `&quot;`)
— как в обычном HTML. `>` экранировать не обязательно, но для симметрии и
читаемости лучше тоже через `&gt;`.

> **cmd.exe (rust-порт):** `<git/>` и `<cmd run="...">` под `--shell cmd`
> вернут явную ошибку, а не "почти работающий" промпт — у `PROMPT` в
> cmd.exe нет механизма выполнить что-то заново при каждой отрисовке (это
> просто статическая строка с `$`-кодами), так что эти теги там
> архитектурно невозможны. На fish/PowerShell/Nushell всё работает как
> обычно.

### Почему это работает "живьём", а не один раз

`<git/>` и `<cmd/>` превращаются не в готовый текст, а в `$(команда)` —
**прямо внутри** итоговой строки `PS1`/`PROMPT`. `psml.py` запускается
один раз и просто пишет шаблон в переменную. А вот сама команда внутри
`$(...)` выполняется шеллом заново **при каждой отрисовке промпта** — это
встроенный механизм bash/zsh (command substitution), он и используется
во всех git-индикаторах типа `__git_ps1`/powerline. Поэтому статус git
всегда актуальный, а не "замороженный" на момент запуска шелла.

Для bash это работает без доп. настроек (опция `promptvars` включена по
умолчанию). Для **zsh** обязательно нужна `setopt PROMPT_SUBST` — если ты
использовал `<git/>` или `<cmd/>`, `psml.py --shell zsh` сам добавит эту
строку в вывод.

## Пробелы и переносы строк

Как и в HTML, отступы и переносы строк в исходном `.psml`-файле — это
просто форматирование, а не часть промпта. Текстовый узел, который **весь**
состоит из пробелов/табов/переводов строк (отступ между тегами на разных
строках), выбрасывается целиком. А пробел внутри одной строки (например,
между `</color>` и `<symbol/>`) — осознанный, он сохраняется.

Если нужен пробел прямо перед закрывающим тегом — поставь его на той же
строке, без переноса после:

```html
<!-- так пробел после $ сохранится -->
<symbol/> </body>

<!-- а так — нет, перевод строки после пробела "съест" весь узел -->
<symbol/>
</body>
```

## Подключение к `.bashrc` / `.zshrc`

### Почему нельзя просто "питон сохраняет переменную"

Питон — отдельный процесс. Переменные окружения наследуются только
**вниз**, от родителя к потомку (шелл → питон), а не наоборот: дочерний
процесс физически не может изменить переменные родительского шелла — это
ограничение модели процессов в ОС, а не питона. Поэтому единственный
способ — пусть **сам шелл** прочитает вывод питона и присвоит его себе.
Для этого и существует `$(...)`/`eval` — команда выполняется в подпроцессе,
а её *вывод* подставляется в команду, которую выполняет уже сам шелл.

### Рекомендуемый способ — `eval`

По умолчанию (без `--raw`) скрипт печатает готовую строку вида
`PS1='...'` (и, если нужно, `setopt PROMPT_SUBST` перед ней) — это можно
просто выполнить через `eval`:

```bash
# ~/.bashrc
eval "$(python3 ~/psml.py)"
```

```bash
# ~/.zshrc
eval "$(python3 ~/psml.py --shell zsh)"
```

Без аргумента скрипт сам берёт `~/ps1.psml` — удобно положить туда основной
файл и не указывать путь каждый раз.

`eval` тут безопасен, потому что выполняется *твой собственный* вывод
твоего собственного скрипта, а не что-то из интернета.

### Альтернатива — без `eval`, через `--raw`

```bash
PS1="$(python3 ~/psml.py --raw)"
```

Тоже работает, но **только если не используешь `<git/>`/`<cmd/>` в zsh** —
этот способ захватывает один кусок вывода в переменную и не выполнит
отдельную строку `setopt PROMPT_SUBST`, которую `psml.py` добавляет в
полном режиме. В bash таких побочных строк не бывает, так что `--raw` там
всегда безопасен.

### Производительность

`python3 ~/psml.py` запускается **один раз** — при старте интерактивной
сессии шелла, когда выполняется `.bashrc`/`.zshrc`. Дальше это просто
строка-шаблон: `\u`, `\h`, `\w` и `$(...)`-вставки шелл подставляет сам, на
лету, без повторного запуска питона при каждой команде или нажатии Enter.

## CLI

```text
python3 psml.py [file] [--shell bash|zsh] [--raw]

  file       путь к .psml, "-" для stdin, или ничего (тогда ~/ps1.psml)
  --shell    bash или zsh (если не указан — берётся из <psml shell="..">)
  --raw      печатать только саму строку приглашения, без "PS1=..."
```
